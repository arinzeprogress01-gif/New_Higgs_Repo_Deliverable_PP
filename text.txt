📄 SYSTEM OVERVIEW (For Your Documentation)

Here’s a clean version you can use:

🏗 System Overview – Chuks Kitchen Backend

Chuks Kitchen is a food ordering backend system that manages:

User registration

Food listing

Cart operations

Order processing

Payment simulation

Stock management

The system is built using:

FastAPI

SQLAlchemy

SQLite (or your DB)

RESTful architecture

🔄 Core Flows
1. Registration Flow

User registers → Account created → Can place orders.

2. Food Management Flow (Admin)

Admin:

Add food

Update price

Update stock

Delete food

3. Cart Flow

User:

Add food to cart

Update quantity

Remove item

Clear cart

Stock not affected at this stage.

4. Ordering Flow

User:

Converts cart → Order

Order status = pending

Total calculated and stored

5. Payment Flow

User:

Pays for pending order

System:

Validates stock

Reduces stock permanently

Creates payment record

Marks order as paid

Clears cart

⚠️ Edge Cases i Handled

Prevent adding quantity above stock

Prevent reducing cart below 1

Prevent double payment

Prevent paying non-existent order

Prevent paying another user’s order

Revalidate stock at payment time

Prevent stock race issues (basic level)

🧠 Assumptions

You should document:

Payment is simulated (no external gateway)

Orders are instantly paid (no cancellation stage)

No timeout for pending orders

Admin manually manages stock

📈 Scalability Thoughts

Replace simulated payment with Stripe/Paystack
